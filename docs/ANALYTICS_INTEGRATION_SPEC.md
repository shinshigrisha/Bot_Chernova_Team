# Спецификация интеграции Telegram AI-бота с аналитикой доставки

Документ описывает требования к источнику данных аналитики (Superset / Grafana) и способам подключения бота Delivery Assistant для сценариев отчётов по мили, SLA, курьерам и ЗиЗ.

---

## 1. Тип источника данных

| Компонент | Рекомендуемый тип | Примечание |
|-----------|-------------------|------------|
| **Superset** | PostgreSQL / Greenplum / ClickHouse / BigQuery | Superset подключается к БД как к Data Source; тип задаётся при создании подключения. |
| **Grafana** | Та же БД или отдельный Data Source | Grafana может использовать тот же PostgreSQL/warehouse для дашбордов. |
| **Бот (читающий слой)** | **PostgreSQL** (предпочтительно) или тот же warehouse | Единый источник правды упрощает согласованность; для бота достаточно read-only подключения. |

**Итог:** если Superset и Grafana уже работают с **PostgreSQL** (или совместимым хранилищем с SQL-интерфейсом), бот может подключаться к той же БД в режиме только чтения. Альтернатива — отдельная аналитическая БД / слой (DWH), куда данные реплицируются или ETL-пайплайном загружаются — тогда тип источника для бота совпадает с типом этого хранилища (часто PostgreSQL или Greenplum).

---

## 2. Параметры подключения

Шаблон переменных окружения для **read-only** подключения бота к аналитической БД:

| Параметр | Переменная окружения | Описание | Пример |
|----------|----------------------|----------|--------|
| **host** | `ANALYTICS_DB_HOST` | Хост сервера БД | `analytics-db.internal` или `localhost` |
| **port** | `ANALYTICS_DB_PORT` | Порт | `5432` |
| **db_name** | `ANALYTICS_DB_NAME` | Имя базы данных | `analytics` или `delivery_dwh` |
| **schema** | `ANALYTICS_DB_SCHEMA` | Схема по умолчанию (опционально) | `delivery` или `public` |
| **user** | `ANALYTICS_DB_USER` | Пользователь с правами только на чтение | `bot_reader` |
| **password** | `ANALYTICS_DB_PASSWORD` | Пароль (хранить в секретах, не в коде) | — |
| **ssl** | `ANALYTICS_DB_SSL` | Включить SSL (`true`/`false`) | `true` для продакшена |
| **vpn / whitelist** | — | Сетевой доступ: либо бот в той же VPN, либо IP бота добавлен в whitelist файрвола/security group БД | Документировать в runbook |

Полный URL для синхронного драйвера (например, для Celery-задач или скриптов):

```text
postgresql://{user}:{password}@{host}:{port}/{db_name}?sslmode=require
```

Для async (если бот будет ходить в analytics напрямую):

```text
postgresql+asyncpg://{user}:{password}@{host}:{port}/{db_name}?ssl=require
```

Рекомендации по безопасности:

- Учётная запись только с правами `SELECT` на нужные схемы/таблицы или представления.
- Не использовать учётку Superset/Grafana для бота; завести отдельного пользователя `bot_reader`.
- SSL — обязательно в продакшене; при необходимости указать путь к CA-сертификату.

## 3. Список таблиц, views и datasets по доставке

Ниже — минимальный набор сущностей, которые должны быть доступны боту для сценариев аналитики (last mile, SLA, курьеры, ЗиЗ, качество).

| Сущность | Тип | Назначение |
|----------|-----|------------|
| Заказы (сырые/агрегированные) | таблица или view | Заказы доставки с полями заказа, ДС, зоны, времени, статуса. |
| Справочник ДС / ВС / ТТ | таблица или view | Маппинг кодов ДС, ВС (зоны), ТТ (торговые точки). |
| ЗиЗ (зоны и здания) | таблица или view | Идентификаторы зон/зданий для разбивки отчётов по ЗиЗ. |
| Курьеры | таблица или view | Связь заказ–курьер, имя/ID курьера для статистики по курьерам. |
| SLA / promised time | поля в заказах или отдельная таблица | Обещанное время доставки (слот/дедлайн) для расчёта нарушений SLA. |
| Delivered time (факт) | поля в заказах | Фактическое время доставки/завершения заказа. |
| Статусы заказов | enum/справочник или поля | Статусы (например «Успешно завершен», «Отменён») для фильтрации и расчёта метрик. |
| Обращения / жалобы | таблица или view | Обращения клиентов или курьеров по заказам (для отчётов по качеству и эскалациям). |

Конкретные имена таблиц/views задаются в вашей аналитической БД; в коде бота используется слой репозиториев/сервисов, который абстрагирует имена и SQL (см. п. 6).

## 4. Источники по доменам

Маппинг доменов к источникам данных (таблицы/views/datasets).

| Домен | Источник (таблица / view / dataset) | Ключевые поля |
|-------|-------------------------------------|----------------|
| **Заказы** | Основная таблица/ view заказов доставки | id заказа, дата/время, ДС, зона, ТТ, статус, promised/delivered time, курьер (см. ниже). |
| **ДС / ВС / ТТ** | Справочник складов/точек или поля в заказах | `ds_code` (ДС), `zone_code` / ВС, `tt_code` или аналог (ТТ). |
| **ЗиЗ** | Справочник или поле в заказах | Идентификатор зоны/здания (например `ziz` или `zone_id`) для группировки. |
| **Курьеры** | Таблица заказов или связь заказ–курьер | `courier_id`, `courier_name` или внешний ключ на справочник курьеров. |
| **SLA / promised time** | Поле в таблице заказов или отдельная таблица слотов | `promised_at`, `deadline_at`, `slot_start`/`slot_end` — для расчёта соблюдения SLA. |
| **Delivered time** | Поле в таблице заказов | `delivered_at`, `finish_at`, `completed_at` — фактическое время завершения. |
| **Статусы** | Поле в заказах + при необходимости справочник | `status`, `order_state`; значения: «Успешно завершен», «Отменён», «Возврат» и т.д. |
| **Обращения / жалобы** | Отдельная таблица или view обращений | id заказа, тип обращения, дата, результат — для отчётов по качеству и эскалациям. |

Если в вашей схеме одна широкая таблица заказов содержит ДС, ТТ, ЗиЗ, курьера, promised/delivered и статус — достаточно одной таблицы (или одного view) с перечисленными полями. Если данные разнесены — нужны соответствующие join’ы, желательно инкапсулированные во views.

## 5. Data dictionary по полям

Поля, которые использует или ожидает бот/доменная модель last mile (см. `app/domain/analytics/mile/calculators.py`, `DeliveryOrderRaw`, `MileOrder`).

### 5.1. Заказ (уровень одной доставки)

| Поле | Тип | Описание |
|------|-----|----------|
| `order_key` / `order_id` | string / UUID | Уникальный идентификатор заказа. |
| `ds_code` | string | Код тёмного склада (ДС). |
| `zone_code` / `vs` | string, nullable | Код зоны / ВС (для ЗиЗ и разбивки по зонам). |
| `tt_code` / `tt` | string | Код торговой точки (ТТ). |
| `courier_id` / `courier_name` | string, nullable | Идентификатор или ФИО курьера. |
| `ziz` | string, nullable | Зона/здание (ЗиЗ) для группировки. |
| `status` | string | Статус заказа (например «Успешно завершен», «Отменён»). |
| `start_delivery_at` | timestamptz, nullable | Начало доставки (или время выезда). |
| `deadline_at` / `promised_at` | timestamptz, nullable | Обещанное время / дедлайн (SLA). |
| `finish_at` / `delivered_at` | timestamptz, nullable | Фактическое время завершения доставки. |
| `delay_flag` | bool / derived | Признак опоздания (факт > обещание). |
| `assembly_delay_flag` | bool / derived | Признак задержки на сборке. |
| `delivery_wait_minutes` | float, nullable | Время ожидания доставки (минуты). |
| `hour_bucket` | string, nullable | Часовой бакет для пиковых часов (например "14:00"). |
| `trip_id` | string, nullable | Идентификатор рейса/маршрута. |
| `replenishment_flag` / `is_dobor` | bool | Добор. |
| `negative_replenishment_flag` | bool | Негативный добор. |

### 5.2. Внутренняя модель бота (уже есть в приложении)

- **delivery_orders_raw** (таблица ingest): `order_key`, `ds_code`, `zone_code`, `start_delivery_at`, `deadline_at`, `finish_at_raw`, `durations` (jsonb), `raw` (jsonb).
- **MileOrder** (доменная модель): `ds`, `vs`, `tt`, `courier_name`, `ziz`, `status`, `delay_flag`, `assembly_delay_flag`, `delivery_wait_minutes`, `hour_bucket`, `trip_id`, `replenishment_flag`, `negative_replenishment_flag`.

Маппинг полей внешней аналитической таблицы/view к полям выше выполняется в коде репозитория или ETL при загрузке в `delivery_orders_raw` / при построении `MileOrder`.

### 5.3. Обращения / жалобы (рекомендуемые поля)

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | UUID / bigint | Идентификатор обращения. |
| `order_id` / `order_key` | string | Связь с заказом. |
| `created_at` | timestamptz | Дата/время обращения. |
| `type` / `category` | string | Тип (жалоба, вопрос, претензия). |
| `resolution` | string, nullable | Исход (закрыто, эскалация и т.д.). |

## 6. Предпочтительный способ интеграции

| Способ | Оценка | Комментарий |
|--------|--------|-------------|
| **Прямой read-only SQL** | Рекомендуется | Один источник правды, минимальная задержка, не нужен отдельный API. Права только на SELECT; предпочтительно читать **views**, а не сырые таблицы. |
| **Views** | Рекомендуется | Отдельные views под сценарии бота (заказы по мили, агрегаты по курьерам, по ЗиЗ) упрощают поддержку и изолируют схему аналитики от бота. |
| **API (REST)** | Допустимо | Если доступ к БД из контура бота невозможен (политика безопасности), можно поднять тонкий read-only API поверх той же БД или Superset API; плюс — контроль формата и лимитов. |
| **Регулярная выгрузка (CSV/файлы)** | Уже есть частично | Ingest в бота из CSV/API уже заложен (`ingest_batches`, `delivery_orders_raw`). Подходит для ежедневных/ежечасовых отчётов с латентностью; для интерактивных запросов из бота предпочтительнее прямой SQL или API. |

**Итоговая рекомендация:**

1. **Основной вариант:** прямой **read-only SQL** к аналитической БД (PostgreSQL или совместимой), доступ только к **views**, созданным для бота (заказы last mile, агрегаты по курьерам/ЗиЗ/SLA).
2. Учётная запись БД — отдельный пользователь с минимальными привилегиями (`SELECT` по нужным схемам/views).
3. SSL — обязательно в продакшене; при необходимости VPN или whitelist IP.
4. Если прямой доступ к БД запрещён — предусмотреть тонкий **read-only API** (или использование Superset API с ограниченными дашбордами/запросами), с последующей подстановкой данных в те же доменные модели бота.

После появления реальных имён таблиц/views и схемы в вашей аналитической БД можно зафиксировать их в конфиге (или в коде репозиториев) и доработать `app/tools/delivery_tools.py` и сервисы аналитики для запросов к этим views.
