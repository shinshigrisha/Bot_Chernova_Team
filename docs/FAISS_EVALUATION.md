# Оценка: нужен ли FAISS для семантического поиска

## 1. Текущий масштаб retrieval

| Источник | Масштаб | Где хранятся векторы | Как ищется |
|----------|---------|----------------------|------------|
| **FAQ** | ~725 записей (seed), в БД — по факту после seed/миграций | PostgreSQL `faq_ai.embedding_vector` (vector 384) | pgvector: HNSW + fallback IVFFlat (миграция 018) |
| **Cases (ML)** | 90 кейсов | Файл `data/ai/ml_cases_embeddings.json`, in-memory в `CaseClassifier` | Линейный проход по словарю (O(n), n=90) |

- **Embeddings**: канонический `EmbeddingsService` (local SentenceTransformer или OpenAI), размерность 384 (MiniLM).
- **FAQ retrieval**: `FAQRepository.search_semantic()` → SQL с `<=> vector_cosine_ops`; индекс HNSW уже ускоряет поиск.
- **Case retrieval**: `CaseClassifier.find_similar_case_semantic()` — один запрос → один вектор, сравнение с ~90 векторами в памяти.

---

## 2. Нужен ли FAISS сейчас?

**Нет, на текущем масштабе FAISS не оправдан.**

Причины:

1. **FAQ (~725 строк)**  
   Поиск уже идёт через **pgvector с HNSW**. Для десятков тысяч векторов pgvector HNSW даёт субсекундный поиск. Добавлять FAISS означало бы дублировать индекс (БД + отдельный индекс на диске/в памяти), усложнять пайплайн пересборки и синхронизацию — без выигрыша по скорости при таком N.

2. **Cases (90 векторов)**  
   Линейный проход по 90 векторам (384 float) — доли миллисекунды. FAISS выгоден при десятках/сотнях тысяч векторов или при очень частом batch-поиске; для 90 записей накладные расходы (загрузка индекса, зависимость, поддержка) не окупаются.

3. **Порог для рассмотрения FAISS в будущем**  
   Имеет смысл вернуться к идее FAISS, если:
   - кейсов станет **порядка 10k+** и поиск по ним станет узким местом, или
   - появится отдельный большой векторный пул (не FAQ в БД), который не хочется держать в PostgreSQL.

---

## 3. Если бы FAISS был нужен (для справки)

Ниже — минимальный вариант без переписывания канонического embedding-сервиса и без замены текущего retrieval.

- **Где использовать**: только для **cases** (in-memory индекс вместо линейного скана), FAQ оставить на pgvector.
- **Минимальный дизайн**:
  - Опциональная зависимость `faiss-cpu` (или `faiss-gpu` при необходимости).
  - Модуль `src/core/services/ai/faiss_case_index.py`: загрузка/построение индекса из `ml_cases_embeddings.json`, метод `search(query_embedding, k=1)` → список (case_id, score).
  - В `CaseClassifier`: при наличии FAISS-индекса вызывать его, иначе — текущий линейный `find_similar_case_semantic`.
- **Файлы для изменений**: `case_classifier.py` (внедрение опционального FAISS), новый `faiss_case_index.py`, `pyproject.toml`/`setup.py` (опциональная зависимость), скрипт пересборки индекса при обновлении `rebuild_case_embeddings.py`.
- **Сборка/пересборка**: после `python -m scripts.rebuild_case_embeddings` — пересоздание FAISS-индекса из обновлённого `ml_cases_embeddings.json`.

Сейчас этот шаг **не делаем**: масштаб не оправдывает внедрение.

---

## 4. Риски (если бы добавляли FAISS)

- Дублирование данных: кейсы в JSON + индекс FAISS — нужна дисциплина пересборки после изменения кейсов/эмбеддингов.
- Доп. зависимость и возможные конфликты окружения (особенно с GPU-версией).
- Два пути поиска (FAQ в БД, cases в FAISS) — чуть сложнее отладки и мониторинга.

## 5. Команды для проверки текущего поведения (без FAISS)

```bash
# Количество FAQ в БД (после миграций и seed)
# В скрипте smoke или админке уже есть faq_count
python -m scripts.smoke_ai   # выведет FAQ_COUNT=...

# Количество кейсов и факт загрузки эмбеддингов
wc -l data/ai/ml_cases.jsonl
# Проверка, что CaseClassifier загружает эмбеддинги (логи при старте: case_classifier_embeddings_loaded, count=90)

# Семантический поиск по FAQ (pgvector)
pytest tests/test_faq_semantic_search.py -v

# Общий smoke retrieval
pytest tests/test_semantic_retrieval_smoke.py -v
```

---

**Итог**: на текущем объёме (FAQ ~725 с pgvector HNSW, cases 90) FAISS не вводим. Оставляем текущую схему; при росте кейсов до 10k+ или появлении нового большого векторного пула — пересмотреть решение и при необходимости добавить FAISS только для этого пула с минимальными изменениями в коде (как в п. 3).
