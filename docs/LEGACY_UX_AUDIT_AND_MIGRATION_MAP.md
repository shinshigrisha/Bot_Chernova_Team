## 3. Best patterns to migrate

- **Единая кнопка «◀️ Назад» в каждом подменю** с явным callback возврата (в legacy — `admin:back_to_main` или контекстный возврат в родительское меню). В текущем — уже есть `build_section_nav_keyboard()` и `with_section_nav()`; распространить на все разделы админки.
- **Описание разделов в одном блоке текста** при открытии админ-панели (как в legacy: краткий список разделов с пояснением). Сейчас при `/admin` только «Админ-панель:» и кнопки; добавить 2–3 строки описания разделов.
- **Иерархия callback:** `admin:<section>_menu` для входа в раздел, `admin:<section>:<action>` для действий, `admin:back_to_main` для возврата. В текущем уже используется `admin:verification_menu` и т.д.; привести именование к единому стилю и документировать.
- **Верификация (админская):** списки с пагинацией и кнопка «Назад» в родительское подменю (в legacy — «◀️ Назад» в `admin:monitoring:verification`). В текущем — один экран pending с кнопками по заявкам; при росте списка стоит добавить пагинацию по образцу legacy.
- **Подтверждение критичных действий:** клавиатура «✅ Подтвердить» / «❌ Отмена» с явными callback (в legacy — `get_confirmation_keyboard(confirm_callback, cancel_callback)`). В текущем уже есть в verification и частично в админке; унифицировать через общий хелпер.
- **Текст при отмене в FSM:** в legacy отмена по «отмена»/«cancel». В текущем — поддержать те же ключевые слова в FSM верификации и админских сценариях и давать короткую подсказку в приветствии/инструкции.

## 4. Patterns to avoid

- **Десятки slash-команд для админов** (legacy: `/setup_ziz`, `/list_groups`, `/add_group`, `/stats`, `/create_polls`, `/set_topic`, `/get_topic_id` и т.д.). В текущем проекте управление — через кнопки; не переносить управление обратно на команды.
- **Один шаг верификации (только ФИ)** из legacy — в текущем проекте верификация осознанно расширена (роль, ТТ, телефон, заявка). Не упрощать до одного поля.
- **Отдельный «Legacy разделы» в меню** и дублирование двух корневых админ-меню (как в неиспользуемом `src/bot/admin/menu.py`). Либо один канонический админ-интерфейс, либо явное «сервисное меню» без смешивания с основным.
- **Смешивание reply- и inline-клавиатур** в одном сценарии. В legacy везде inline; в текущем тоже. Сохранять единый стиль (inline для меню и выбора).
- **Жёсткая привязка к опросам в корневом UX** (расписание опросов, слоты, голосования). По заданию опросы — низкий приоритет; не выносить их в главное меню или обязательные экраны.

## 5. Proposed unified menu architecture

- **Корень (entrypoint):**  
  `/start` → по статусу/роли один экран: guest → регистрация; pending/rejected/blocked → свои экраны; approved → главное меню по роли (админ / курьер / куратор). Без лишних slash-команд в обычном UX.

- **Префиксы callback:**  
  - `root:*` — корневое меню (main, verification, admin, help, ai_curator).  
  - `nav:*` — общая навигация (back, main, cancel, help).  
  - `admin:*` — админ-панель: `admin:back_to_main`, `admin:<section>_menu`, `admin:<section>:<action>[:id]`.  
  - `verification:*` — пользовательская верификация (start, role, confirm).  
  - Ролевые: `courier:*`, `curator:*` — для пунктов меню курьера/куратора.

- **Админ-панель:**  
  - Один главный экран с текстом-описанием разделов и кнопками.  
  - Каждый раздел — свой экран с кнопкой «◀️ Назад» (`admin:back_to_main` или возврат в подменю, например `admin:monitoring_menu`).  
  - В каждом разделе внизу — единый блок навигации: **Назад** | **Главное меню** (nav:main) и при необходимости **Отмена** / **Помощь**.

- **Section page layout:**  
  - Заголовок раздела (кратко).  
  - Текст/список (при необходимости пагинация).  
  - Кнопки действий раздела.  
  - Последний ряд: навигация (Назад, Главное меню, при необходимости Отмена/Помощь).

- **Верификация (пользователь):**  
  - Вход только через кнопку «Регистрация» (root/verification).  
  - Многошаговый FSM с возможностью отмены по «отмена»/«cancel».  
  - После подачи заявки — экран ожидания; админам — уведомление с кнопками Одобрить/Отклонить/Заблокировать (текущее поведение сохранить и усилить в задаче 3).

## 6. Safe migration order

1. **Документация и константы (без изменения поведения)**  
   - Зафиксировать в коде или в docs единый style guide callback (префиксы, суффиксы `_menu` для разделов).  
   - Удалить или явно пометить мёртвый код: `src/bot/admin/menu.py` (legacy keyboard и `legacy_root`), если не планируется использовать.

2. **Админ-главный экран**  
   - Добавить краткое описание разделов в сообщение при открытии админ-панели (по образцу legacy).  
   - Убедиться, что везде, где открывается раздел, есть «◀️ Назад» с `admin:back_to_main` или контекстным возвратом.

3. **Навигация в разделах**  
   - Во всех админ-разделах использовать `with_section_nav()` (или эквивалент) с единым набором: Назад, Главное меню, при необходимости Отмена.  
   - Добавить подсказку про «отмена»/«cancel» в FSM верификации и админских потоков.

4. **Верификация (админ)**  
   - При появлении большого числа заявок — добавить пагинацию экрана pending (по образцу legacy `admin:verification:verify:page:N`).  
   - Сохранить текущие кнопки Одобрить/Отклонить и расширить до «Заблокировать»/«Открыть карточку» по задаче 3.

5. **Разделы админки по задачам 4–5**  
   - Подключать реальные разделы (Верификация, AI-куратор, FAQ, Анализ CSV, Мониторинг, Рассылка, ТМЦ) по мере реализации, с единым layout и навигацией.

Опросы (polls) не включать в приоритет миграции UX.
