Ниже готовая разбивка **на задачи, этапы и промпты**, чтобы можно было прямо запускать в работу.

Я опираюсь на два набора правил:
модуль **аналитики мили** с обязательными метриками, причинами просрочки, анализом ДС → ВС → ТТ, ЗиЗ, курьеров, пиков и спасённых заказов  и модуль **аналитики качества по обращениям** с разбором тегов, повреждений, возвратов, проблемных дарксторов, ЗиЗ и курьеров .

---

# 1. Общая цель проекта

Собрать Telegram AI-бота, который:

* читает подготовленные данные доставки
* отвечает на вопросы по мили и качеству
* делает автоотчёты
* находит проблемные ДС, ВС, ТТ, ЗиЗ и курьеров
* показывает причины и рекомендации
* работает быстро за счёт своих агрегатов, а не через UI Superset/Grafana

* есть ли views / datasets / API

### Задача A2. Получить технические параметры доступа

Нужно получить:

* тип БД
* host
* port

* schema
* login
* password

### Задача A3. Получить список таблиц или views

Минимально нужны источники по:

* заказам

* факту доставки
* обращениям / жалобам

Минимум:

* `fact_delivery_orders`

## Блок C. ETL и нормализация

### Задача C1. Загрузка данных из источника

Варианты:

* прямой SQL из аналитической БД
* views
* API
* временно CSV/JSON выгрузка

### Задача C2. Очистка и нормализация полей

Нужно привести к единому виду:

* даты
* статусы
* названия ТТ
* ЗиЗ
* курьеров
* сервисы

### Задача C3. Отфильтровать валидные записи

По правилам мили анализ проводится только по заказам со статусом **«Успешно завершен»**, а источник просрочки определяется только по полю **«Доставка в срок»**.

## Блок D. Расчёт бизнес-метрик

### Задача D1. Метрики мили

Нужно считать:

* число заказов
* число успешных
* просрочки
* просрочки > 15 минут
* пиковые часы
* среднее ожидание доставки
* заказов на курьера
* спасённые заказы
* проблемные рейсы
* доборы
* негативные доборы
* проблемные ЗиЗ
* главную причину просрочки

Эти расчёты прямо требуются в правилах мили. 

### Задача D2. Метрики качества

Нужно считать:

* количество обращений
* обращения на 100 заказов
* повреждения
* возвраты
* нарушения по тегам
* проблемные дарксторы
* проблемные ЗиЗ
* проблемных курьеров

Это требуется правилами анализа обращений. 

### Задача D3. Логика выбора главной причины просрочки

Причина должна быть одной из пяти:

* сборка
* дефицит курьеров
* плохая маршрутизация
* перегруженные ЗиЗ
* смешанная проблема

* проблемные курьеры

## Блок F. AI-слой

### Задача F1. Маршрутизация запросов

Разделить:

* `mile_analytics`
* `delivery_quality_analytics`

### Задача F2. System prompt и правила ответа

Нужно внедрить уже подготовленные правила.

### Задача F3. Форматирование ответа

Для мили правила требуют:

* сначала общая информация по кусту
* затем топ спасённых курьеров
* потом ДС сверху вниз
* потом ВС
* каждый ТТ отдельным блоком
* использовать эмодзи и Telegram-friendly формат 

Для обращений формат должен быть структурирован:

* вывод
* таблица
* причины
* рекомендации

Сравнить результаты бота с Superset/Grafana на нескольких датах.

### Задача H2. Проверка логики причин

Проверить, что бот не путает:

* сборку
* дефицит курьеров
* перегруженность ЗиЗ

Бот не должен:

# 8. Самая практичная последовательность на старт

Сначала делайте **mile_analytics**, потому что там больше обязательных операционных сценариев: проблемные ДС/ЗиЗ/курьеры, пики, дефицит, перегруз, спасённые заказы, сервисы и причины просрочки. Это подробно зафиксировано в правилах мили. 

После этого подключайте **delivery_quality_analytics** по обращениям: теги, повреждения, возвраты, проблемные точки и рекомендации.
