# A. Новый пользователь

### 1. `/start`

Пользователь видит:

```
Добро пожаловать в Delivery Assistant

Для доступа необходимо пройти регистрацию.
```

Кнопка:

```
📋 Подать заявку
```

---

### 2. Регистрация

Бот последовательно спрашивает:

1️⃣ Роль

* Курьер
* Куратор

2️⃣ Имя

3️⃣ Фамилия

4️⃣ ТТ

5️⃣ Телефон

После заполнения:

```
Ваша заявка отправлена на проверку.
Ожидайте решения администратора.
```

Статус пользователя:

```
pending
```

# B. Администратор

### После подачи заявки

Админ **моментально** получает сообщение:

```
Новая заявка на доступ

Фамилия: Петров
Роль: Курьер
ТТ: 7367
Телефон: +799999999
TG ID: 12345678
```

```
✅ Одобрить
❌ Отклонить
⛔ Заблокировать
```

# D. AI-куратор

При нажатии:

```

```

```
Опишите проблему с доставкой.

Например:
— покупатель не отвечает
— разбил товар
— недовоз
— не проходит оплата
— опаздываю
```

Быстрые кнопки:

```
📞 Клиент не отвечает
🥚 Разбил товар
📦 Недовоз
💳 Не проходит оплата
⏱ Опаздываю
📍 Проблема с адресом
✍️ Другое
```

# E. Ответы AI

AI должен отвечать **коротко и по делу**.

Пример:

### Пользователь

```
клиент не отвечает
```

### Ответ

```
Порядок действий:

1. Позвоните клиенту 3 раза.
2. Напишите сообщение в чат заказа.
3. Подождите 5 минут.

Если ответа нет — свяжитесь с куратором.
```

---

### Другой пример

```
разбил яйца
```

Ответ:

```
Повреждение товара.

1. Зафиксируйте фото.
2. Сообщите куратору.
3. Укажите повреждение в комментарии заказа.
```

---

# F. AI не должен давать тупые ответы

Плохой ответ:

```
Опишите подробнее.
```

Если кейс очевидный.

# 1. Общая цель проекта

Собрать Telegram AI-бота, который:

* читает подготовленные данные доставки
* отвечает на вопросы по мили и качеству
* делает автоотчёты
* находит проблемные ДС, ВС, ТТ, ЗиЗ и курьеров
* показывает причины и рекомендации
* работает быстро за счёт своих агрегатов, а не через UI Superset/Grafana

# 3. Разбивка на конкретные задачи

## Блок A. Доступ к данным

### Задача A1. Выяснить реальный источник данных

Нужно узнать:

* какая БД стоит под Superset
* какая БД стоит под Grafana
* есть ли прямой read-only доступ
* есть ли views / datasets / API

### Задача A2. Получить технические параметры доступа

Нужно получить:

* тип БД
* host
* port
* db_name
* schema
* login
* password
* ssl / vpn / whitelist

### Задача A3. Получить список таблиц или views

Минимально нужны источники по:

* заказам
* курьерам
* ДС / ВС / ТТ
* ЗиЗ
* SLA / времени обещания
* факту доставки
* обращениям / жалобам

---

## Блок B. Хранилище AI-бота

### Задача B1. Создать отдельную схему

Например:

* `ai_delivery`

### Задача B2. Создать факт-таблицы

Минимум:

* `fact_delivery_orders`
* `fact_delivery_complaints`

### Задача B3. Создать агрегаты

Минимум:

* `agg_ds_daily`
* `agg_ds_hourly`
* `agg_ziz_daily`
* `agg_courier_daily`
* `agg_tt_daily`
* `agg_quality_daily`

---

## Блок C. ETL и нормализация

### Задача C1. Загрузка данных из источника

Варианты:

* прямой SQL из аналитической БД
* views
* API
* временно CSV/JSON выгрузка

### Задача C2. Очистка и нормализация полей

Нужно привести к единому виду:

* даты
* статусы
* названия ТТ
* ЗиЗ
* курьеров
* сервисы

### Задача C3. Отфильтровать валидные записи

По правилам мили анализ проводится только по заказам со статусом **«Успешно завершен»**, а источник просрочки определяется только по полю **«Доставка в срок»**. 

---

## Блок D. Расчёт бизнес-метрик

### Задача D1. Метрики мили

Нужно считать:

* число заказов
* число успешных
* просрочки
* просрочки > 15 минут
* пиковые часы
* среднее ожидание доставки
* заказов на курьера
* спасённые заказы
* проблемные рейсы
* доборы
* негативные доборы
* проблемные ЗиЗ
* главную причину просрочки

Эти расчёты прямо требуются в правилах мили. 

### Задача D2. Метрики качества

Нужно считать:

* количество обращений
* обращения на 100 заказов
* повреждения
* возвраты
* нарушения по тегам
* проблемные дарксторы
* проблемные ЗиЗ
* проблемных курьеров

Это требуется правилами анализа обращений. 

### Задача D3. Логика выбора главной причины просрочки

Причина должна быть одной из пяти:

* сборка
* дефицит курьеров
* плохая маршрутизация
* перегруженные ЗиЗ
* смешанная проблема 

---

## Блок E. Backend API

### Задача E1. API под мили

Эндпоинты:

* просрочки по ДС
* топ курьеров
* проблемные ЗиЗ
* пиковые часы
* анализ ТТ
* спасённые заказы
* SLA-тревоги

### Задача E2. API под качество

Эндпоинты:

* жалобы по периоду
* анализ тегов
* проблемные дарксторы
* проблемные курьеры
* обращения по ЗиЗ / ТТ / ДС

### Задача E3. Безопасный слой запросов

Backend должен использовать только:

* заранее подготовленные таблицы
* разрешённые SQL-шаблоны
* параметры даты / ДС / ТТ / курьера

---

## Блок F. AI-слой

### Задача F1. Маршрутизация запросов

Разделить:

* `mile_analytics`
* `delivery_quality_analytics`

### Задача F2. System prompt и правила ответа

Нужно внедрить уже подготовленные правила.

### Задача F3. Форматирование ответа

Для мили правила требуют:

* сначала общая информация по кусту
* затем топ спасённых курьеров
* потом ДС сверху вниз
* потом ВС
* каждый ТТ отдельным блоком
* использовать эмодзи и Telegram-friendly формат 

Для обращений формат должен быть структурирован:

* вывод
* таблица
* причины
* рекомендации 

---

## Блок G. Telegram

### Задача G1. Подключить команды

Минимум:

* просрочки
* топ курьеров
* проблемные ЗиЗ
* пики
* жалобы
* полный анализ

### Задача G2. Автоотчёты

Сценарии:

* утренний отчёт

* тревога SLA

### Задача G3. Удобный текст ответа

---

## Блок H. Проверка качества

### Задача H1. Сверка цифр

Сравнить результаты бота с Superset/Grafana на нескольких датах.

### Задача H2. Проверка логики причин

Проверить, что бот не путает:

* сборку
* дефицит курьеров
* перегруженность ЗиЗ
* маршрутизацию

### Задача H3. Проверка запретов

Бот не должен:

* пропускать ТТ
* объединять ТТ

* делать выводы без данных
