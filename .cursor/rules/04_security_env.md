# Rule 04: Безопасность и env-секреты

## Цель
Исключить утечки секретов и персональных данных при разработке, логировании и эксплуатации.

## Секреты и конфигурация

1. Все ключи и секреты хранить только в переменных окружения.
   - API keys, токены, DSN с паролями, webhook secret и т.д.
   - Не хардкодить секреты в коде, тестах, документации, migration-файлах.

2. `.env` — локальный файл разработчика.
   - Не коммитить реальные значения в git.
   - Использовать `.env.example` только с безопасными плейсхолдерами.

3. Для новых интеграций:
   - сразу добавлять переменные в конфиг и `.env.example`;
   - не подставлять fallback-ключи по умолчанию.

## Логирование

1. Никогда не логировать API keys и токены.
   - Запрещено выводить секреты даже в debug-режиме.
   - При необходимости логировать только факт наличия ключа (`configured=true/false`) без значения.

2. Не логировать PII (персональные данные).
   - Примеры: ФИО, телефоны, email, адреса, документы, точные координаты, полный текст личной переписки.
   - Если данные критичны для трассировки, использовать маскирование/редакцию.

3. Сообщения об ошибках:
   - не должны содержать секреты из исключений или request payload;
   - при сериализации исключений очищать чувствительные поля.

## Работа с внешними API

1. Не отправлять лишние пользовательские данные внешним провайдерам.
   - Передавать только минимальный необходимый контекст.
   - Для AI-запросов исключать PII, если это не обязательно для задачи.

2. Учитывать принцип least privilege:
   - использовать ключи с минимальными правами;
   - разделять ключи по окружениям (dev/stage/prod).

## Чеклист перед merge

- Новые секреты вынесены в env.
- В коде и логах нет утечек ключей.
- PII не сохраняется в логи.
- `.env.example` обновлен плейсхолдерами при необходимости.
