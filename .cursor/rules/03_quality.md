# Rule 03: Качество изменений

## Цель
Любое изменение должно быть проверяемым, воспроизводимым и безопасным для текущего поведения системы.

## Обязательные проверки

Для каждого изменения запускать доступные в проекте проверки:

1. `mypy` — проверка типов;
2. `ruff` — линтинг/стиль;
3. `pytest` — тесты.

Если один из инструментов не настроен в проекте:
- явно зафиксировать это в отчете по задаче;
- не подменять отсутствующую проверку "на глаз".

## Минимальные требования к новому модулю

Любой новый модуль обязан иметь минимум один smoke-test:

- проверяет критичный happy-path;
- запускается локально без сложной ручной подготовки;
- фиксирует базовую работоспособность (инициализация, вызов ключевой функции, ожидаемый результат/маршрут).

Примеры допустимого smoke-теста:
- отдельный скрипт в `scripts/`;
- простой `pytest`-тест с минимальными моками;
- golden-case прогон для AI-маршрута.

## Правила изменения кода

1. Не вносить "частично готовый" код:
   - без TODO-заглушек в критичных ветках;
   - без неиспользуемых публичных интерфейсов.

2. После изменения:
   - проверить импорты и отсутствие циклических зависимостей;
   - проверить, что логика fallback/ошибок не стала молчаливой;
   - убедиться, что команды/роуты не конфликтуют с существующими.

3. Для AI-флоу:
   - подтверждать, что при отсутствии внешних ключей/сервисов система деградирует корректно;
   - не допускать "тишины" вместо ответа.

## Требования к отчету по задаче

В отчете указывать:

- какие проверки запускались;
- что прошло/не прошло;
- где добавлен smoke-test и как его запустить;
- какие ограничения/риски остались.
